## Description

Decompose the Meridian Pay monolith (`meridian-api`, 340K LOC, Java 17) into domain-aligned microservices. The monolith was built in 2021 for a $5M ARR business — it's now buckling under $18M ARR load. Symptoms: deployment queue averaging 3.2 days, 6 production incidents in Q4 2025 (vs. 2 in Q4 2024), and test suite takes 47 minutes.

We're not doing a big-bang rewrite. The strategy is strangler fig: extract one bounded context at a time, use CDC (Debezium + Kafka) for data synchronization, and maintain the monolith as the system of record until each service proves stable in production.

Phase 1 (Q1): Extract `payment-service` and `merchant-service`
Phase 2 (Q2): Extract `settlement-service`, `notification-service`, and `reporting-service`
Phase 3 (Q3): Decommission monolith routing layer

## Technical Approach

- **Strangler Fig Pattern**: New services sit behind an API gateway (Kong); traffic is gradually shifted per-endpoint
- **Data Strategy**: CDC via Debezium streams PostgreSQL WAL to Kafka topics; new services consume events and maintain local read stores
- **Service Mesh**: Istio for mTLS, circuit breaking, and traffic management
- **Database per Service**: Each extracted service gets its own PostgreSQL schema (same RDS instance initially, separate instances by Q3)
- **API Gateway**: Kong deployed in EKS, handles routing, rate limiting, auth token validation
