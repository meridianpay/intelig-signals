## Sprint Goals

### Goal 1: Deploy fraud detection model in shadow mode
**Status**: ACHIEVED

Deploy the XGBoost fraud detection model alongside the existing rules engine in shadow mode. The model scores every transaction but doesn't block — results are logged for comparison. Target: model must demonstrate 40% improvement in fraud detection rate over rules engine on live traffic.

**Outcome**: Shadow mode deployed Feb 26. After 10 days of live traffic (2.3M transactions scored), the ML model detected 847 fraudulent transactions that the rules engine missed. False positive rate: 0.22% (target: < 0.3%). Fraud detection improvement: 52% over rules engine. Ready for production cutover in Sprint 30.

### Goal 2: Payment-service extraction to staging
**Status**: ACHIEVED

Complete the extraction of the payment-service from the monolith and deploy to staging. CDC pipeline via Debezium must be operational with < 500ms replication lag. All payment processing endpoints must pass integration test suite against the new service.

**Outcome**: Payment-service deployed to staging Feb 28. CDC pipeline operational with P99 replication lag of 180ms. 247 of 251 integration tests passing — 4 failures related to edge cases in refund processing (fixed Mar 3). Zero data inconsistencies detected in 7-day soak test.

### Goal 3: Complete merchant onboarding automation (INI-203)
**Status**: ACHIEVED

Ship the automated merchant onboarding flow to production. KYB via Middesk, risk scoring, contract generation via PandaDoc, and self-serve provisioning must all be operational.

**Outcome**: Shipped to production Mar 1. First 12 merchants onboarded through the new flow with average time of 1.2 business days (target: 2 days). Operations team confirmed 34 hours/week savings. Initiative INI-203 marked COMPLETED on Mar 4.

## Retrospective

### What Went Well
- **Fraud model performance exceeded expectations**: 52% improvement over rules engine in shadow mode is significantly better than the 40% target. The team's investment in feature engineering (147 features) paid off.
- **CDC pipeline stability**: Debezium + Kafka setup has been rock solid after initial configuration. P99 lag of 180ms gives us confidence for production cutover.
- **INI-203 early completion**: Merchant onboarding shipped 3 days ahead of schedule. Operations team reaction was immediate and positive — they started redirecting time to merchant success within 48 hours.
- **Cross-team collaboration**: Ravi's fraud team and David's decomposition team coordinated well on the internal API contract for transaction metadata.

### What Didn't Go Well
- **Settlement reports slipped**: Dependency on design system v2 wasn't identified early enough. The settlement reports UI needs components that aren't in the current design system. Pushed to Sprint 30.
- **PCI work underestimated**: HSM integration is more complex than anticipated. The Thales SDK documentation is poor, and we spent 2 days on a configuration issue that turned out to be a firmware version mismatch.
- **Capacity planning miss**: We allocated 94 points but only delivered 89. The 5-point miss was split between PCI (2 points) and monolith decomposition (2 points), plus 1 point of unplanned tech debt.
- **Duplicate email incident**: Should have caught the idempotency gap in code review. Adding webhook idempotency to our PR checklist.

### Changes for Next Sprint
1. **Add HSM integration buffer**: Allocate 2 extra points for PCI work to account for Thales SDK complexity
2. **Design system dependency tracking**: Fatima to meet with design team weekly to surface blockers earlier
3. **Webhook idempotency checklist**: All external webhook integrations must implement idempotency checks (added to PR template)
4. **CDC monitoring**: Ship the monitoring dashboard in Sprint 30 before payment-service goes to production traffic
